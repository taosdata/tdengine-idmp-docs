# Pipeline

Parsing, extraction prep, AI extraction, merging, and failure handling.

## Stage 1: Section Parsing (`parse.py`)

Pure Python, no AI. Reads every in-scope doc, splits into sections, writes the `.sections/` directory and `.sections/manifest.yaml`.

### Parsing rules

1. Apply the inclusion/exclusion policy from `02-scope-and-definitions.md` to filter files
2. Split each file on H2 (`##`) and H3 (`###`) headings
3. Each section includes: the heading text + all body content until the next heading of equal or higher level
4. The file's frontmatter (`title`, `sidebar_label`) becomes metadata for the top-level section
5. Content before the first H2 is captured as the intro section with `heading: "(intro)"` and `heading_level: 0`
6. Sections with no meaningful content (e.g., only a `<DocCardList />` component) are skipped
7. Generate `section_id` per the rules in `03-data-contracts.md`
8. Compute `content_hash` (SHA-256 of heading line + body)

### Handling existing extraction results

`parse.py` regenerates `section.md` and `meta.yaml` for every section on every run, but it **preserves existing `extraction.yaml` files** for sections whose `content_hash` has not changed. For sections whose content has changed (different `content_hash`), `parse.py` deletes the stale `extraction.yaml` so the AI agent will reprocess them.

Sections that no longer exist (deleted from docs) have their entire directory removed, including any `extraction.yaml`.

### Output

A directory tree under `.sections/` (git-ignored). Each section gets its own directory named by the `section_id` components:

```
.sections/
├── elements/
│   ├── creating-elements/
│   │   ├── section.md          # raw section content (heading + body)
│   │   └── meta.yaml           # section_id, heading_path, content_hash, source file path
│   ├── element-templates/
│   │   ├── section.md
│   │   └── meta.yaml
│   └── (intro)/
│       ├── section.md
│       └── meta.yaml
├── data-modeling/index/     # disambiguated (index.md collision)
│   └── overview/
│       ├── section.md
│       └── meta.yaml
└── visualization/index/     # disambiguated (index.md collision)
    └── overview/
        ├── section.md
        └── meta.yaml
```

The directory structure maps directly to `section_id`: `elements/creating-elements/` → `elements#creating-elements`. For disambiguated collisions: `data-modeling/index/overview/` → `data-modeling/index#overview`.

Each `meta.yaml` contains:

```yaml
section_id: "elements#creating-elements"
file: "03-data-modeling/01-elements.md"         # full relative path for link generation
heading: "Creating Elements"
heading_level: 2
anchor: "creating-elements"                     # URL fragment for deep links; null for intro sections
heading_path: ["Data Modeling", "Elements", "Creating Elements"]
content_hash: "a3f2b8c4..."
```

Why one directory per section instead of a monolithic YAML:
- **AI extraction can process individual files** — read one `section.md`, extract capabilities, no need to parse a giant YAML
- **Incremental updates are trivial** — changed content produces a changed file; file modification time or hash diff tells you what needs re-extraction
- **Parallelizable** — multiple extraction calls can process different sections concurrently
- **Inspectable** — developers can browse individual sections during debugging
- **Git-ignored** — `.sections/` is regenerated by `parse.py`, so it doesn't bloat the repo

The `.sections/` directory also contains a manifest file:

```
.sections/manifest.yaml         # list of all section_ids with content_hashes, for quick diffing
```

This file is deterministic: running `parse.py` twice on the same docs produces identical output.

## Stage 2: Extraction Prep (`prepare.py`)

Pure Python, no AI. Reports extraction status and writes the AI prompt file. **`prepare.py` never writes to `capabilities.section-map.yaml`** — that file is owned exclusively by `merge.py`.

### What it does

`prepare.py` compares `.sections/manifest.yaml` against the existing section map to report which sections need extraction. It also handles prompt version changes:

1. Load `.sections/manifest.yaml` and `capabilities.section-map.yaml` (if any)
2. Identify sections needing extraction: those without an `extraction.yaml`, or whose `content_hash` doesn't match the section map (stale `extraction.yaml` files are already deleted by `parse.py` — see Stage 1)
3. If `prompt_version` in the section map differs from the current prompt template, delete **all** `extraction.yaml` files and rewrite the prompt (effectively a full re-extraction)

Note: `parse.py` (Stage 1) already handles deletion of stale `extraction.yaml` files for changed/deleted sections. `prepare.py`'s primary value is the status report and prompt version management.

### Output

1. **`.sections/extraction-prompt.md`** — a short, generic prompt file placed at the root of `.sections/`. This is persistent infrastructure: `prepare.py` writes it on the first run and leaves it in place for all subsequent runs. It does **not** embed any section content — it tells the AI agent how to find and process sections.

The prompt file is simple and stable:

````
# Capability Extraction Task

For each section directory under `.sections/` that contains a `section.md`
but no `extraction.yaml`, do the following:

1. Read `section.md`.
2. Identify capabilities — things the system can do: tasks, functions, or
   behaviors available to users or administrators.
   Examples: "trend-chart", "element-search", "sso-authentication".
   Do NOT treat navigation instructions, UI layout descriptions, or
   references to external systems as capabilities.
3. For each capability, classify the relation:
   - "defined" — this section explains what the capability is, how it works,
     or how to use/configure it.
   - "referenced" — this section meaningfully depends on or integrates with
     the capability, but does not define it.
   Do NOT mark incidental mentions (navigation links, see-also footnotes)
   as referenced.
4. Write `extraction.yaml` in the same directory as `section.md`.

Output format for each `extraction.yaml`:

```yaml
capabilities:
  - id: "short-lowercase-hyphenated-id"
    relation: "defined"     # or "referenced"
    confidence: "high"      # or "medium" or "low"
```

If a section contains no capabilities:

```yaml
capabilities: []
```
````

2. **Summary to stdout** — reports what it found. Example:

```
Sections: 487 total, 12 need extraction, 475 up to date
```

## Stage 3: AI Extraction (Manual — AI Agent)

The operator directs an AI agent (e.g., Claude Code) to process the sections identified in Stage 2. This is a manual step — the operator decides when and how to invoke the agent.

### Workflow

1. Direct the AI agent to read `.sections/extraction-prompt.md`
2. The agent walks `.sections/`, reads each `section.md` that has no `extraction.yaml`, and writes `extraction.yaml` in the same directory
3. Run `merge.py` to collect all `extraction.yaml` files and merge into the section map

The AI agent uses its native file-reading and file-writing tools. Each section is self-contained — the agent reads one `section.md`, writes one `extraction.yaml`. No batching, no combined output files.

### Why per-section extraction?

- **Simple prompt** — no embedded content, no combined output format to get right
- **Naturally incremental** — sections that already have `extraction.yaml` are skipped
- **Partial completion is free** — if the agent times out or hits context limits, whatever sections were processed are done. Run the agent again and it picks up where it left off.
- **Inspectable** — each `extraction.yaml` sits next to its `section.md`, easy to review individually

### `merge.py` — Result Collection & Merge

`merge.py` is the **sole writer** of `capabilities.section-map.yaml`. It rebuilds the section map from scratch on every run, using `.sections/manifest.yaml` as the authoritative list of sections and collecting extraction results from `extraction.yaml` files.

Steps:

1. **Load `.sections/manifest.yaml`** — the authoritative list of all current sections with their `content_hash` values
2. **Load existing `capabilities.section-map.yaml`** (if any) — used to carry forward results for sections that weren't re-extracted
3. **For each section in the manifest:**
   a. If an `extraction.yaml` exists in the section directory **and** the adjacent `meta.yaml` `content_hash` matches the manifest entry → validate, normalize, and use this extraction result
   b. If no `extraction.yaml` exists but the section map has an entry with a **matching `content_hash`** → carry forward the existing section map entry (extraction is still current)
   c. If a section map entry exists at a **different `section_id`** but with the same `content_hash` → remap (section was moved/renamed). Carry forward under the new `section_id`.
   d. Otherwise → section has no extraction result yet (new or changed). Include in the section map with `capabilities: []` and `extracted_at: null` to signal it still needs extraction.
4. **Sections in the old section map but not in the manifest** → deleted. Drop them (log a warning if they had `defined` capabilities).
5. **Validate each `extraction.yaml`** — check YAML is valid, has expected schema
6. **Normalize capability IDs** — lowercase, trim whitespace, replace spaces with hyphens
7. **Deduplicate** — remove duplicate capability IDs within a single section
8. **Enrich with section metadata** — copy `file`, `heading`, `heading_level`, and `anchor` from the adjacent `meta.yaml` into each section map entry, so the committed file is self-contained
9. **Write atomically** — write to temp file, then rename

If any `extraction.yaml` is malformed, `merge.py` reports the error (with the section directory path) and skips that file. The section falls through to carry-forward (step 3b) or empty (step 3d) as appropriate.

**Safety rule:** `merge.py` only merges an `extraction.yaml` if its adjacent `meta.yaml` `content_hash` matches the manifest. This prevents stale or orphaned extraction files (left behind by filesystem edits or parser changes) from being merged accidentally.

After a successful merge, `prepare.py --check` can verify whether any sections still need extraction.

Dependencies: `pyyaml` only

## Taxonomy Bootstrap (First Run Only)

After the first full extraction and merge, `prepare.py --bootstrap` produces a draft `capabilities.taxonomy.yaml`:

1. Collect all unique capability IDs with `relation: defined` and `confidence: high` or `medium`
2. Cluster similar IDs using string similarity (e.g., `trend-chart-creation` and `trend-chart-configuration`)
3. For each cluster, propose a canonical ID and name
4. Infer categories from the source file's chapter directory
5. Parse `20-roadmap/index.md` to add `planned` entries
6. Write draft taxonomy for human review

## Output: What Consumers Get

The pipeline produces two committed files that together form the complete capability inventory:

| File | Owner | Contains |
|---|---|---|
| `capabilities.section-map.yaml` | Machine (AI agent + merge) | Which capabilities each doc section defines or references, with confidence levels |
| `capabilities.taxonomy.yaml` | Human curator | Canonical capability names, categories, status, aliases, parent/child relationships, ignored list |

There is no generated "final output" file. Any report, web app, or other consumer reads both files and joins them. The join logic is straightforward:

### How to consume the data

1. **Load the section map** — gives you a list of `(section_id, capability_id, relation, confidence)` tuples
2. **Load the taxonomy** — gives you canonical capability definitions with `id`, `name`, `category`, `status`, `since`, `aliases`, `parent`, and the `ignored` list
3. **Join extracted IDs to canonical capabilities:**
   - Direct match on taxonomy `id` → mapped
   - Match on any taxonomy `aliases` entry → mapped to the capability that owns that alias
   - Match on `ignored` list → skip
   - No match → unmatched (flag for human triage)
4. **For each canonical capability**, collect its `defined` and `referenced` sections. Each section map entry includes `file`, `heading`, `heading_level`, and `anchor` — sufficient to generate deep links (`{file_url}#{anchor}`) and group by source file.
5. **Derive `children`** from `parent` fields in the taxonomy
6. **Filter by confidence** as appropriate for your use case (e.g., exclude `low` confidence from a customer-facing report, include everything in an internal audit)

This keeps the pipeline focused on data extraction and curation, while giving consumers full flexibility over presentation, filtering, and formatting.

## Failure Handling

### Malformed AI output

If the AI agent writes an invalid `extraction.yaml` for a section, `merge.py` reports the error (with the directory path), skips that section, and continues merging valid ones. The operator can:
- Fix the malformed `extraction.yaml` manually and re-run `merge.py`
- Delete the bad `extraction.yaml` and re-run the AI agent (it will reprocess sections without an `extraction.yaml`)

### Partial extraction

Since each section is extracted independently, partial completion is handled naturally:
- Sections with a valid `extraction.yaml` are merged
- Sections without one are left for the next extraction run
- The operator can simply re-run the AI agent — it skips sections that already have `extraction.yaml`

### Stale extraction results

Two layers of protection:
1. **`parse.py`** deletes `extraction.yaml` for any section whose `content_hash` has changed (see Stage 1)
2. **`merge.py`** independently verifies `content_hash` consistency and refuses to merge an `extraction.yaml` whose adjacent `meta.yaml` `content_hash` doesn't match the manifest

### Section map recovery

If the section map file is corrupted or deleted:
- Run `parse.py` then `prepare.py --force` — this deletes all `extraction.yaml` files, forcing a full re-extraction
- Run the AI agent, then `merge.py` — `merge.py` rebuilds the section map from scratch
- The taxonomy file is unaffected since it's independently maintained.
