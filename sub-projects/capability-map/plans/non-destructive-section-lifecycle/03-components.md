# Solution Components

One existing artifact is extended; one new artifact is introduced.

## `extraction.yaml` — extended with `content_hash`

**Owner:** AI extraction agent (writes), `prepare.py` (reads for staleness)
**Location:** `.sections/<name_path>/<slug>/extraction.yaml`

A `content_hash` field is added as the first key, recording the hash of the section content
at the time of extraction. This makes the file self-describing: any reader can determine
whether the extraction is current without consulting any other file.

Schema (extended):
```yaml
content_hash: abc123def    # SHA-256 prefix of section.md at extraction time; matches meta.yaml
capabilities:
  - id: trend-chart
    relation: defined
    confidence: high
  - id: element-search
    relation: referenced
    confidence: medium
```

The `content_hash` value must match the `content_hash` in `meta.yaml` for the extraction
to be considered current. This is the same hash that `merge.py` already validates via
`meta.yaml` — the stamp in `extraction.yaml` allows `prepare.py` to perform the same check
earlier in the pipeline.

**Backward compatibility:** Existing `extraction.yaml` files without a `content_hash` field
are treated as stale by `prepare.py` (missing hash = needs re-extraction). `merge.py`'s
existing validation is unaffected since it reads the hash from `meta.yaml`, not `extraction.yaml`.

**Agent responsibility:** The AI agent must write the `content_hash` field when it creates
`extraction.yaml`. The value to use is available in `meta.yaml` in the same directory.

## `.sections/needs-extraction.yaml`

**Owner:** `prepare.py` (written each run outside `--check` mode)
**Read by:** AI extraction agent

Lists section IDs that need extraction — either because `extraction.yaml` is absent, or
because its `content_hash` does not match the manifest.

Schema:
```yaml
# Generated by prepare.py — sections that need capability extraction.
generated_at: '2026-03-21'
total: 42
sections:
  - introduction#what-is-tdengine-idmp
  - data-modeling/index#overview
```

Lifecycle:
- Written (or overwritten) by every non-`--check` `prepare.py` run
- `sections: []` when nothing needs extraction (clears stale state from prior run)
- On `--force`, lists all sections (all `extraction.yaml` files are deleted first)

**Note — derivability:** The content of this file is fully derivable by scanning `.sections/`
and comparing `content_hash` in each `extraction.yaml` against the sibling `meta.yaml`. At
700+ sections this scan is fast (well under a second). The file exists not because the
information is hard to compute, but to keep the AI agent's job simple: it reads a flat list
rather than understanding the pipeline's staleness logic or directory structure. If the
agent were ever replaced by a script that could do the comparison itself, this file could
be dropped without any loss of correctness.
