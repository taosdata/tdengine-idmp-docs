# Follow-On Work: Promoting Pipeline Artifacts to Git

## Context

`.sections/` is currently git-ignored in its entirety because it is treated as a
fully-regenerated scratch space. This is appropriate for most of its contents (703+
directories of `section.md`, `meta.yaml`, `extraction.yaml`), but two files in that
directory have a different character and belong in source control.

This work is not part of the section lifecycle problem, but it is a natural next step
once the lifecycle plan is implemented.

## Files to Promote

### `extraction-prompt.md`

**Current location:** `.sections/extraction-prompt.md` (git-ignored, generated by `prepare.py`)

**Problem:** `extraction-prompt.md` is authored content — it defines how the AI agent
behaves. It changes intentionally when the extraction logic evolves, not as a byproduct of
parsing. Keeping it out of git means:
- Changes to agent instructions are invisible in the commit history
- Reviewing or rolling back a prompt change requires inspecting the Python string in `prepare.py`
- The `write_prompt_if_needed()` mechanism exists specifically because the file can be lost

**Proposal:** Move to a permanent, committed location — e.g., `scripts/extraction-prompt.md`
or the project root. Update `prepare.py` to read from (not write to) this location. The
`EXTRACTION_PROMPT_CONTENT` constant and `write_prompt_if_needed()` function would be removed;
the agent simply reads the file directly. The prompt version mechanism could be simplified
or replaced with standard git history.

### `manifest.yaml`

**Current location:** `.sections/manifest.yaml` (git-ignored, generated by `parse.py`)

**Problem:** `manifest.yaml` is the authoritative index of all in-scope sections and their
content hashes. It is read by `prepare.py`, `merge.py`, and `validate.py`. Keeping it
out of git means:
- You cannot `git diff` to see which sections were added, removed, or changed between doc
  versions — this is useful context alongside any commit that touches docs
- There is no historical record of when a section was introduced or when its content hash changed
- If `.sections/` is accidentally deleted, `parse.py` must re-run before any other script works

**Proposal:** Move to a committed location — e.g., the project root alongside
`capabilities.section-map.yaml`. Update the `SECTIONS_DIR / "manifest.yaml"` path reference
in `shared.py` and all scripts. `parse.py` continues to regenerate it on every run; the
difference is that the result is now committed.

## Section Last-Modified Timestamp in `meta.yaml`

**Current state:** `meta.yaml` records the `content_hash` of the section but not when
it last changed. Knowing when a section was last modified is useful for auditing,
prioritisation, and debugging (e.g. "which sections changed in the last week?").

**Proposal:** Add a `last_modified` field to `meta.yaml`, written by `parse.py` whenever
the section's `content_hash` changes from the previous run:

```yaml
section_id: data-modeling/index#overview
content_hash: abc123def
last_modified: '2026-03-21'   # ISO date; updated only when content_hash changes
...
```

This lifecycle plan removes the old-meta read from `parse.py` (the stale extraction check),
so implementing `last_modified` requires reintroducing a read of the existing `meta.yaml`
before writing the new one. `parse.py` already writes `meta.yaml` for every section; adding
a read immediately before that write is a small, self-contained change. Carry forward
`last_modified` unchanged if the `content_hash` is the same; set it to today's date if the
hash changed or the file is new.

**Scope:** Small, self-contained change to `parse.py` and the `meta.yaml` schema. No
downstream scripts read `last_modified` today, so there is no breakage risk. The field
would also be useful context for the AI agent (e.g. to deprioritise recently-changed
sections that are likely still in flux).

**Dependency:** None — can be done independently of this plan or the git-promotion work.

## What Not to Promote

The following artifact introduced by the lifecycle plan should remain git-ignored:

- `.sections/needs-extraction.yaml` — transient work queue, changes every `prepare.py` run

This file reflects pipeline state at a point in time, not stable authored or derived content.
(Note: the lifecycle plan explicitly rejected a `.deleted/` holding area and `needs-deletion.yaml` — those artifacts do not exist.)

## Scoping and Dependencies

This work is independent of the lifecycle plan and can be done before or after it.
However, doing it after is cleaner because:
- The lifecycle plan changes `EXTRACTION_PROMPT_CONTENT` — promoting the file after avoids
  a two-step update to both the Python string and the committed file
- The lifecycle plan updates how `prepare.py` uses the manifest — any path changes are
  easier to make once the surrounding code is settled

## Implementation Sketch

1. Move `extraction-prompt.md` to its permanent location; update `.gitignore` accordingly
2. Update `prepare.py` to reference the new path; remove `write_prompt_if_needed()` and
   `EXTRACTION_PROMPT_CONTENT`; update `EXTRACTION_PROMPT_FILE` constant in `shared.py`
3. Move `manifest.yaml` path reference in `shared.py` to the new committed location;
   update all scripts that reference `SECTIONS_DIR / "manifest.yaml"`
4. Add both files to git and commit
5. Update `architecture/pipeline.md` file ownership table and data-flow description
6. Update `guides/running-the-pipeline.md` if it references the `.sections/` paths
