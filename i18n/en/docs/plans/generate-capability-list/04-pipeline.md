# Pipeline

Parsing, extraction prep, AI extraction, merging, and failure handling.

## Stage 1: Section Parsing (`parse.py`)

Pure Python, no AI. Reads every in-scope doc, splits into sections, writes `capabilities.sections.yaml`.

### Parsing rules

1. Apply the inclusion/exclusion policy from `02-scope-and-definitions.md` to filter files
2. Split each file on H2 (`##`) and H3 (`###`) headings
3. Each section includes: the heading text + all body content until the next heading of equal or higher level
4. The file's frontmatter (`title`, `sidebar_label`) becomes metadata for the top-level section
5. Content before the first H2 is captured as the intro section with `heading: "(intro)"` and `heading_level: 0`
6. Sections with no meaningful content (e.g., only a `<DocCardList />` component) are skipped
7. Generate `section_id` per the rules in `03-data-contracts.md`
8. Compute `content_hash` (SHA-256 of heading line + body)

### Output

A directory tree under `.sections/` (git-ignored). Each section gets its own directory named by the `section_id` components:

```
.sections/
├── elements/
│   ├── creating-elements/
│   │   ├── section.md          # raw section content (heading + body)
│   │   └── meta.yaml           # section_id, heading_path, content_hash, source file path
│   ├── element-templates/
│   │   ├── section.md
│   │   └── meta.yaml
│   └── (intro)/
│       ├── section.md
│       └── meta.yaml
├── data-modeling/index/     # disambiguated (index.md collision)
│   └── overview/
│       ├── section.md
│       └── meta.yaml
└── visualization/index/     # disambiguated (index.md collision)
    └── overview/
        ├── section.md
        └── meta.yaml
```

The directory structure maps directly to `section_id`: `elements/creating-elements/` → `elements#creating-elements`. For disambiguated collisions: `data-modeling/index/overview/` → `data-modeling/index#overview`.

Each `meta.yaml` contains:

```yaml
section_id: "elements#creating-elements"
file: "03-data-modeling/01-elements.md"         # full relative path for link generation
heading: "## Creating Elements"
heading_level: 2
heading_path: ["Data Modeling", "Elements", "Creating Elements"]
content_hash: "a3f2b8c4..."
```

Why one directory per section instead of a monolithic YAML:
- **AI extraction can process individual files** — read one `section.md`, extract capabilities, no need to parse a giant YAML
- **Incremental updates are trivial** — changed content produces a changed file; file modification time or hash diff tells you what needs re-extraction
- **Parallelizable** — multiple extraction calls can process different sections concurrently
- **Inspectable** — developers can browse individual sections during debugging
- **Git-ignored** — `.sections/` is regenerated by `parse.py`, so it doesn't bloat the repo

The `.sections/` directory also contains a manifest file:

```
.sections/manifest.yaml         # list of all section_ids with content_hashes, for quick diffing
```

This file is deterministic: running `parse.py` twice on the same docs produces identical output.

## Stage 2: Extraction Prep (`prepare.py`)

Pure Python, no AI. Compares `.sections/manifest.yaml` against the existing section map, determines which sections need (re-)extraction, and produces a ready-to-use prompt file for the AI agent.

### Change detection

On every run, `prepare.py`:

1. Load `.sections/manifest.yaml` (just reparsed by `parse.py`)
2. Load existing `capabilities.section-map.yaml` (if any — first run starts with empty section map)
3. For each section in the manifest:
   - **Same `section_id` and same `content_hash` as section map** → unchanged, carry forward cached result
   - **Same `section_id`, different `content_hash`** → content changed, queue for extraction
   - **New `section_id`** (not in section map) → new section, queue for extraction
4. For each section in section map not in the manifest:
   - **`content_hash` found at a different `section_id`** → section was renamed/moved. Carry forward the cached result under the new `section_id`, remove old entry. No extraction needed.
   - **`content_hash` not found anywhere** → section was deleted. Remove from section map. Log warning if it had `defined` capabilities.
5. If `prompt_version` in the section map differs from the current prompt template, **all** sections are queued (effectively a full run).

On first run (no section map file exists), all sections are queued — this is just the natural outcome of step 3, not a special mode.

Note: directory moves and numeric prefix changes don't affect `section_id` (it uses the stripped filename only), so reorganizing chapters doesn't create orphaned section map entries. Only meaningful filename renames or heading changes produce a new `section_id`.

Every content change triggers extraction, regardless of diff size. A one-word change can alter capability semantics. The cost of extracting one section is trivial compared to the risk of stale data.

### Output

`prepare.py` writes:

1. **Updated `capabilities.section-map.yaml`** — with unchanged entries carried forward, moved entries remapped, deleted entries removed. Sections queued for extraction are **not** in this file (they have no cached result yet).

2. **`.sections/extraction-prompt.md`** — a self-contained prompt file that the AI agent can process directly. Contains the extraction instructions, the section content for all queued sections (grouped by source file for context), and the expected output format. If no sections need extraction, this file is not created and `prepare.py` reports "section map is up to date."

The prompt file follows this structure:

````
# Capability Extraction Task

You are analyzing sections of TDengine IDMP documentation to identify system capabilities.

A capability is something the system can do — a task, function, or behavior
available to users or administrators.
Examples: "trend-chart", "element-search", "sso-authentication".

Do NOT treat the following as capabilities:
- Navigation instructions (e.g., "click the menu")
- UI layout descriptions that aren't functional
- References to external systems or standards

For each capability you identify, classify the relation:
- "defined" — this section explains what the capability is, how it works,
  or how to use/configure it. The section is a primary source for this capability.
- "referenced" — this section meaningfully points to, depends on, or
  integrates with the capability, but does not define it.
  Example: "Anomaly detection can trigger event creation" in an Events section
  is a meaningful reference to anomaly-detection.

Do NOT mark a capability as referenced for incidental mentions, e.g.,
a capability appearing in a list of navigation links or a see-also footnote.

## Sections to analyze

### File: 03-data-modeling/01-elements.md

#### Section: elements#creating-elements

```markdown
## Creating Elements
...section content...
```

#### Section: elements#element-templates

```markdown
## Element Templates
...section content...
```

### File: 04-visualization/02-chart-types/01-trend-chart.md

...more sections...

## Required output format

For each section, output a YAML list entry keyed by its section_id.

```yaml
- section_id: "{section_id}"
  capabilities:
    - id: "{short-lowercase-hyphenated-id}"
      relation: "defined"    # or "referenced"
      confidence: "high"     # or "medium" or "low"
```

If a section contains no capabilities, output:

```yaml
- section_id: "{section_id}"
  capabilities: []
```

You MUST output exactly one entry per section_id provided, in the same order.

Write the output to `capabilities.extraction-result.yaml`.
````

3. **Summary to stdout** — reports how many sections are unchanged, queued, moved, and deleted. Example:

```
Sections: 487 total, 12 queued for extraction, 470 cached, 3 moved, 2 deleted
Prompt file written to: .sections/extraction-prompt.md
```

## Stage 3: AI Extraction (Manual — AI Agent)

The operator directs an AI agent (e.g., Claude Code) to process the prompt file produced by Stage 2. This is a manual step — the operator decides when and how to invoke the agent.

### Workflow

1. Open the prompt file: `.sections/extraction-prompt.md`
2. Direct the AI agent to read it, process the sections, and write the result to `capabilities.extraction-result.yaml`
3. Run `merge.py` to validate the result and merge it into the section map

### `merge.py` — Result Validation & Merge

After the AI agent produces `capabilities.extraction-result.yaml`, `merge.py` validates and merges it into the section map:

1. **Validate response shape** — confirm every `section_id` from the prompt file has a corresponding output entry
2. **Normalize capability IDs** — lowercase, trim whitespace, replace spaces with hyphens
3. **Deduplicate** — remove duplicate capability IDs within a single section
4. **Retain confidence** — keep all confidence levels in the section map (consumers decide how to filter)
5. **Merge** — combine validated results with the existing `capabilities.section-map.yaml` (which `prepare.py` already updated with carried-forward and moved entries)
6. **Write atomically** — write to temp file, then rename
7. **Clean up** — delete `capabilities.extraction-result.yaml` and `.sections/extraction-prompt.md` on success

If validation fails (missing sections, invalid YAML), `merge.py` reports the errors and exits without modifying the section map. The operator can then re-run the AI agent or fix the result file manually.

Dependencies: `pyyaml` only

## Taxonomy Bootstrap (First Run Only)

After the first full extraction and merge, `prepare.py --bootstrap` produces a draft `capabilities.taxonomy.yaml`:

1. Collect all unique capability IDs with `relation: defined` and `confidence: high` or `medium`
2. Cluster similar IDs using string similarity (e.g., `trend-chart-creation` and `trend-chart-configuration`)
3. For each cluster, propose a canonical ID and name
4. Infer categories from the source file's chapter directory
5. Parse `20-roadmap/index.md` to add `planned` entries
6. Write draft taxonomy for human review

## Output: What Consumers Get

The pipeline produces two committed files that together form the complete capability inventory:

| File | Owner | Contains |
|---|---|---|
| `capabilities.section-map.yaml` | Machine (AI agent + merge) | Which capabilities each doc section defines or references, with confidence levels |
| `capabilities.taxonomy.yaml` | Human curator | Canonical capability names, categories, status, aliases, parent/child relationships, ignored list |

There is no generated "final output" file. Any report, web app, or other consumer reads both files and joins them. The join logic is straightforward:

### How to consume the data

1. **Load the section map** — gives you a list of `(section_id, capability_id, relation, confidence)` tuples
2. **Load the taxonomy** — gives you canonical capability definitions with `id`, `name`, `category`, `status`, `since`, `aliases`, `parent`, and the `ignored` list
3. **Join extracted IDs to canonical capabilities:**
   - Direct match on taxonomy `id` → mapped
   - Match on any taxonomy `aliases` entry → mapped to the capability that owns that alias
   - Match on `ignored` list → skip
   - No match → unmatched (flag for human triage)
4. **For each canonical capability**, collect its `defined` and `referenced` sections. Use `.sections/` `meta.yaml` files (or the manifest) to resolve `section_id` → file path, heading, and link.
5. **Derive `children`** from `parent` fields in the taxonomy
6. **Filter by confidence** as appropriate for your use case (e.g., exclude `low` confidence from a customer-facing report, include everything in an internal audit)

This keeps the pipeline focused on data extraction and curation, while giving consumers full flexibility over presentation, filtering, and formatting.

## Failure Handling

### Malformed AI output

If the AI agent produces invalid output, `merge.py` rejects it without modifying the section map:
- **Missing `section_id`s** — reports which sections are missing from the result
- **Invalid YAML** — reports parse errors with line numbers
- **Extra or unexpected fields** — ignored; only known fields are processed

The operator can then re-run the AI agent with the same prompt file, or manually fix `capabilities.extraction-result.yaml` before re-running `merge.py`.

### Partial extraction

If the AI agent only processes some sections (e.g., context limits, session timeout):
- The operator can manually split the prompt file and run extraction in batches
- Each partial result can be merged separately — `merge.py` only updates sections present in the result file, leaving the rest of the section map intact
- Sections not yet in the section map will appear in the next `prepare.py` run as still needing extraction

### Section map recovery

If the section map file is corrupted or deleted:
- Run `parse.py` then `prepare.py` — with no section map, every section is queued. Then run the AI agent and `merge.py` as usual.
- The taxonomy file is unaffected since it's independently maintained.
